# NeuroFabric Manifest (`neurofabric.json`)

This document defines the schema for the `neurofabric.json` manifest, generated by NeuroGenesis during the bootstrap phase (Phase 6: The Lobotomy & Routing Enforcement).

This manifest acts as the definitive configuration for the NeuroFabric microkernel, dividing the Lead Agent's capabilities into "Two Lanes" and enforcing the **Cryptographic Delegation Receipt** protocol.

---

## 1. The Two-Lane Capability Model

To solve the "Hallucination Bypass" problem and prevent Time-of-Check to Time-of-Use (TOCTOU) attacks, the NeuroFabric kernel enforces a strict, mathematically bound execution environment. NeuroGenesis compiles the project's specific risk profile and critical areas into this manifest.

### The Schema

```json
{
  "project_name": "med-tracker-pro",
  "kernel_version": ">=1.2.0",
  "routing": {
    "default_agent": "Lead Agent",
    "capability_pools": {
      "security-analysis": ["arc7-security-sentinel"],
      "financial-audit": ["arc7-financial-auditor"]
    }
  },
  "capabilities": {
    "standard": [
      "nf_read_file",
      "nf_search_code",
      "nf_list_directories"
    ],
    "gated": [
      {
        "tool": "nf_write_file",
        "conditions": [
          {
            "path_pattern": "src/auth/**",
            "required_authority": "security-analysis",
            "rationale": "Modifications to authentication require Security Sentinel approval."
          },
          {
            "path_pattern": "src/finance/reports/**",
            "required_authority": "financial-audit",
            "rationale": "Financial report generation requires Financial Auditor approval."
          }
        ]
      },
      {
        "tool": "nf_execute_bash",
        "conditions": [
          {
            "command_pattern": "^(npm run build|docker build).*",
            "required_authority": "hitl",
            "rationale": "Production builds require Human-in-the-Loop (HITL) approval."
          }
        ]
      }
    ]
  }
}
```

## 2. Kernel Enforcement Mechanics

When the Lead Agent attempts to execute a tool, the NeuroFabric Kernel processes the request through the `neurofabric.json` ruleset:

### Step 1: Lane Routing
*   If the tool is in the `standard` array, the kernel executes it immediately.
*   If the tool is in the `gated` array, the kernel checks the `conditions` (e.g., regex matching the file path or bash command). If a condition matches, it halts execution and issues a Challenge Nonce.

### Step 2: The Action Hash (Anti-Forgery & TOCTOU Prevention)
The kernel does not just ask for a generic "approval". It demands a receipt mathematically bound to the exact action the Lead Agent is trying to perform.
1.  The Lead Agent submits a JSON-RPC payload (e.g., `{"file": "src/auth/login.ts", "content": "..."}`).
2.  The kernel computes the SHA-256 `action_hash` of this exact, canonicalized JSON payload.
3.  The Lead Agent must pass this `action_hash` to the required authority (e.g., the Security Sentinel).

### Step 3: Receipt Validation
When the specialist approves the action, the kernel generates an HMAC receipt binding the `action_hash` to the `challenge_nonce`.
*   **Anti-Forgery:** If the Lead Agent alters even a single character of the code it wants to write after getting approval, the `action_hash` changes, the HMAC signature breaks, and the kernel rejects the execution.
*   **Anti-Replay:** The kernel maintains a fast TTL-evicted cache of consumed nonces. Once the Lead Agent executes the write, the receipt is burned. It cannot be used to overwrite the file a second time without a fresh review.
